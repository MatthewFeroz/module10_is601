# Module 10 — Secure FastAPI Users with PostgreSQL and CI/CD

[![CI/CD](https://github.com/MatthewFeroz/calculator-sql/actions/workflows/test.yml/badge.svg)](https://github.com/MatthewFeroz/calculator-sql/actions/workflows/test.yml)

This project extends the Module 9 FastAPI calculator with a secure,
database-backed user foundation. SQLAlchemy maps user objects to PostgreSQL,
Pydantic validates and normalizes account data, bcrypt hashes every password
with a unique salt, and GitHub Actions tests, scans, and publishes the Docker
image.

- GitHub repository: [MatthewFeroz/calculator-sql](https://github.com/MatthewFeroz/calculator-sql)
- Docker Hub repository: [matthewferoz/calculator-sql](https://hub.docker.com/r/matthewferoz/calculator-sql)
- Reflection: [docs/reflection.md](docs/reflection.md)
- Submission evidence checklist: [docs/evidence/README.md](docs/evidence/README.md)

## Security and Data Design

The `users` table contains the following columns:

| Column | Purpose |
| --- | --- |
| `id` | Integer primary key generated by the database |
| `username` | Required, normalized, indexed, and unique |
| `email` | Required, validated, normalized, indexed, and unique |
| `password_hash` | Salted bcrypt hash; no plaintext password column exists |
| `created_at` | Database-generated, timezone-aware creation timestamp |

The layers have separate responsibilities:

1. `UserCreate` rejects malformed emails, unsafe usernames, weak passwords, and
   values beyond bcrypt's 72-byte input limit.
2. `create_user` checks for friendly duplicate errors and hashes the password
   before constructing the SQLAlchemy model.
3. PostgreSQL unique constraints remain the final protection against concurrent
   duplicate requests.
4. `UserRead` serializes ORM objects without a password or password-hash field.

Public user routes are intentionally deferred because the course introduces
routes in a later module; the model, validation, hashing, and persistence layers
are ready for those routes.

## Run the Application with Docker Compose

Prerequisites: Docker Desktop with Docker Compose.

```bash
docker compose up --build
```

After the services become healthy:

- Application: <http://localhost:8000>
- Interactive API docs: <http://localhost:8000/docs>
- Health check: <http://localhost:8000/health>
- pgAdmin: <http://localhost:5050> (`admin@example.com` / `admin`)
- PostgreSQL: `localhost:5432`, database `fastapi_db`, user/password
  `postgres` / `postgres`

The application initializes missing SQLAlchemy tables during startup. If a
Module 9 volume already contains the older `users` table without
`password_hash`, recreate the coursework volumes before the first Module 10
run:

```bash
# Warning: this intentionally deletes local PostgreSQL and pgAdmin volume data.
docker compose down --volumes
docker compose up --build
```

## Run Tests Locally

### 1. Create the Python environment

Windows PowerShell:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
playwright install chromium
```

macOS/Linux:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
playwright install chromium
```

### 2. Run fast tests without PostgreSQL

```bash
pytest -m "not postgres"
```

This runs the calculator, schema, hashing, API, and browser tests. PostgreSQL
tests are explicitly skipped unless `TEST_DATABASE_URL` is defined, preventing
accidental cleanup of a development database.

### 3. Run the complete suite with real PostgreSQL

Start a disposable PostgreSQL 17 test container:

```bash
docker run --rm --detach --name calculator-sql-test-db \
  --publish 5433:5432 \
  --env POSTGRES_USER=postgres \
  --env POSTGRES_PASSWORD=postgres \
  --env POSTGRES_DB=calculator_test \
  postgres:17
```

Set the dedicated URL and run all 84 tests.

Windows PowerShell:

```powershell
$env:TEST_DATABASE_URL = "postgresql+psycopg2://postgres:postgres@localhost:5433/calculator_test"
pytest
docker stop calculator-sql-test-db
```

macOS/Linux:

```bash
export TEST_DATABASE_URL="postgresql+psycopg2://postgres:postgres@localhost:5433/calculator_test"
pytest
docker stop calculator-sql-test-db
```

The test configuration reports line coverage for `app/` and `main.py`. The
current complete suite passes 84 tests with 100% application coverage.

Useful targeted commands:

```bash
pytest tests/unit/ -v
pytest tests/integration/ -v
pytest tests/e2e/ -v
pytest -m postgres -v
```

## Build and Run the Production Image

```bash
docker build --tag calculator-sql:local .
docker run --rm --publish 8000:8000 \
  --env DATABASE_URL="sqlite+pysqlite:///:memory:" \
  calculator-sql:local
```

The image runs as an unprivileged Linux user and includes an HTTP health check.
To run against PostgreSQL, replace `DATABASE_URL` with a reachable PostgreSQL
SQLAlchemy URL.

After the CI deployment succeeds, pull the published image with:

```bash
docker pull matthewferoz/calculator-sql:latest
docker run --rm --publish 8000:8000 \
  --env DATABASE_URL="sqlite+pysqlite:///:memory:" \
  matthewferoz/calculator-sql:latest
```

## CI/CD Pipeline

`.github/workflows/test.yml` runs for pushes and pull requests to `main`:

1. **test** starts PostgreSQL 17, installs Chromium, runs all tests, and uploads
   JUnit and XML coverage reports.
2. **security** builds the image, performs a live `/health` smoke test, and fails
   on fixable high or critical Trivy findings.
3. **deploy** runs only for successful pushes to `main`, then publishes Linux
   AMD64 and ARM64 images as both `latest` and the immutable Git commit SHA.

All external actions are pinned to full commit SHAs. The Docker image is not
published until both testing and security jobs succeed.

### Required GitHub configuration

Create a Docker Hub repository named `calculator-sql`, then add these GitHub
repository secrets under **Settings → Secrets and variables → Actions**:

| Secret | Value |
| --- | --- |
| `DOCKERHUB_USERNAME` | Docker Hub account name (for this README: `matthewferoz`) |
| `DOCKERHUB_TOKEN` | Docker Hub access token with read/write permission |

The deploy job targets the GitHub environment named `production`. It can be
created under **Settings → Environments** if deployment approval or restricted
secrets are desired.
